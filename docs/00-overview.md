# 00. アドサーバとは何か

> 一言でいうと、**「リクエストに対して、その瞬間に最適な広告を選んで返すサーバ」** のこと。

---

## 「広告が表示されるまで」を分解する

ユーザーが Web ページを開いたときに広告が出る。あの瞬間、裏では何が起きているのか？

```
[ User Browser ]                  [ Publisher (記事サイト) ]      [ Ad Server ]
       |                                    |                            |
       | 1. GET /article                    |                            |
       | -------------------------------->  |                            |
       |    HTML 返す（中に <script> あり）  |                            |
       | <--------------------------------  |                            |
       |                                    |                            |
       | 2. <script> を実行して             |                            |
       |    Ad Server に ad request         |                            |
       | --------------------------------------------------------------> |
       |                                    |                            |
       |                                    |       3. 在庫から広告選定 |
       |                                    |          ターゲティング適用 |
       |                                    |          頻度制御 / 予算   |
       |                                    |                            |
       |    広告 (HTML / 画像 URL / クリック先) を返す                   |
       | <-------------------------------------------------------------- |
       |                                    |                            |
       | 4. 画像を表示                       |                            |
       |    impression pixel を叩く          |                            |
       | --------------------------------------------------------------> |
       |                                    |                            |
       | 5. クリックすると                   |                            |
       |    click redirect 経由で LP へ      |                            |
       | --------------------------------------------------------------> |
```

ポイントは **Publisher の HTML には広告そのものは入っていない** こと。広告タグ（JS スニペット）だけが入っていて、ブラウザが Ad Server に問い合わせる。これが「アドサーバ」が独立したシステムとして存在する理由です。

---

## アドサーバが解く問題

単に「広告を返す」だけなら静的ファイルでも十分です。アドサーバが必要になるのは、次の問題を **1 つのリクエスト内で同時に** 解かなくてはならないから。

| 問題 | 例 |
|------|-----|
| **どれを出す？** | 在庫の中で適切なターゲティング条件にマッチするもの |
| **誰に出す？** | 同じユーザーに 1 日 100 回出してはいけない（frequency cap） |
| **いくらで売る？** | 入札額が高い順 / eCPM 最大化 |
| **予算は？** | 日次予算を使い切ったキャンペーンは止める |
| **計測は？** | impression / click を後で集計できるように記録 |
| **速度は？** | 一般に **100ms 以下** で返さないとページ表示を遅らせる |

## アドサーバの 2 大カテゴリ

| カテゴリ | 立ち位置 | 例 |
|---------|---------|-----|
| **Publisher-side Ad Server** | サイト運営者側で広告枠を管理 | Google Ad Manager |
| **Advertiser-side Ad Server** | 広告主側で配信先を選び入札 | DV360, The Trade Desk |

この教材では **Publisher-side** から始めて、Step 7 で **Advertiser-side（DSP）** に近い実装にも触れます。詳しい登場人物は [01. アドテクエコシステム](01-ecosystem.md) で。

---

## このリポジトリでの最終形

Step 8 までで以下を備えた最小のアドサーバを構築します：

- 在庫管理（DB に Campaign / LineItem / Creative）
- ターゲティング（地域 / デバイス）
- インプレッション / クリックの計測
- フリークエンシーキャップ（Redis）
- 簡易 RTB（OpenRTB 風 bid request, 第二価格）
- 日次予算と簡易ペーシング
- 集計レポート
