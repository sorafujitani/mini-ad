# 07. ペーシングと予算

> 「広告主から預かった 100 万円を、24 時間で**均等に**消化しなさい」

これがペーシング問題のすべて。
言葉にすると簡単だが、imp は非同期に発生し続け、配信プロセスは複数台、というのが厄介。

---

## ペーシングしないとどうなる

予算 10 万円 / 日のキャンペーンを、何の制御もなしに配信すると…

```
00:00 - 01:00 で 10 万円使い切って停止
```

朝の元気な時間帯だけ全部消化して、夕方以降は何も出ない。Advertiser が嫌がる典型ケース。

---

## 2 つの基本戦略

| 戦略 | 動作 | 用途 |
|------|------|------|
| **ASAP** | 制限なく入札・配信。早期に予算消化 | キャンペーン終了間際、即時配信 |
| **Even (Smooth)** | 1 日を通じて均等に消化 | ブランディング、長期キャンペーン |

Even の素朴な実装:

```
1 時間ごとに使うべき額 = daily_budget / 24
当該時間で既に使った額  = D
   D >= 期待消化 → 当該時間は配信を弾く
   D <  期待消化 → 配信 OK
```

ただし「**1 時間境界で配信が再開される現象**」が起きる (グラフがガタガタになる)。
これを解消するのが次の Probabilistic Pacing。

---

## Probabilistic Pacing

「配信するかどうか」を確率で決める方式。

```
pacing_rate = 期待ペース / 実際ペース
            (0 ≤ rate ≤ 1)
配信判定: rand() < pacing_rate なら配信する
```

- 予算を使いすぎている (rate 小) → 各 imp で配信スキップ確率が上がる → 自然に減速
- 予算を残しすぎている (rate ≈ 1) → ほぼ常に配信

これだとガタガタにならず滑らかになる。

---

## 分散環境での予算管理

アドサーバが 1 台ならローカルで予算カウントすれば済むが、複数台になると共有カウンタが必要。

```
[Worker 1]                    [Worker 2]                  [Worker 3]
   │                              │                            │
   └──── DECR redis:budget:c1 ─── ┴────────────────────────────┘
```

実装パターン:

| パターン | 仕組み | 欠点 |
|----------|--------|------|
| **同期 DECR** | imp ごとに Redis DECR | Redis が SPOF / レイテンシボトルネック |
| **Budget Bucket** | 各 Worker に小さな予算を配り、使い切ったら再要求 | バケツ単位で粒度誤差 |
| **Eventual / Lossy** | 各 Worker でローカル消化、N 秒ごとに集約 | 予算超過のリスク |

Google や Facebook 規模だと **数 ms の差で数千円損する** ので、ここはかなり職人芸の世界。

---

## 配信ペーシングと入札ペーシング

ペーシングには 2 つの位置がある:

1. **配信ペーシング**: 出すか出さないかの選別
2. **入札ペーシング**: 出すなら **いくらで** 入れるか

DSP では (2) が主流。**Adaptive Bidding** とも呼び、強化学習で動的に入札額を調整する研究領域でもあります。

---

## 予算階層

| 予算粒度 | 意味 |
|----------|------|
| **総予算** | キャンペーン全期間の上限 |
| **日次予算** | 1 日の上限 |
| **時間予算** | 当該時間内の上限 (ペーシング由来) |

アドサーバは imp ごとに 3 つすべてをチェックする必要がある。
